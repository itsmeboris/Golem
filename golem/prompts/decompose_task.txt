Your job is to break this task into 2-6 concrete, actionable subtasks.

## Codebase Structure

The ``golem`` Python package is laid out as:
- ``golem/core/`` — infrastructure (config, CLI wrapper, dashboard, APIs, notifications)
- ``golem/`` (top-level modules) — orchestrator, supervisor, flow, validation, merge, worktree, prompts
- ``golem/tests/`` — mirrors source structure (``test_<module>.py``)
- ``golem/prompts/`` — prompt template files loaded by ``prompts.py``

Respect module boundaries when splitting subtasks. Group closely related changes
in the same module into a single subtask rather than splitting them.

Produce a JSON response with this structure:
```json
{{
  "subtasks": [
    {{
      "subject": "Short imperative description of subtask 1",
      "description": "Detailed description of what needs to be done"
    }},
    {{
      "subject": "Short imperative description of subtask 2",
      "description": "Detailed description of what needs to be done"
    }}
  ]
}}
```

Guidelines:
- Each subtask should be independently executable by a Claude agent
- Order subtasks logically (dependencies first)
- Keep subjects concise (under 80 characters)
- Include enough detail in descriptions for an agent to work autonomously
- If the task involves code changes, include relevant file paths or module names
- If the task involves investigation, specify what to look for and where
- Prefer fewer subtasks (2-3) for simple tasks (single-file changes, adding a function + tests)
- Only use 4+ subtasks for genuinely complex work (multi-module features, cross-cutting changes)
- Do NOT create separate subtasks for closely related functions in the same file — group them
- The last subtask should always be verification (run tests, lint, check correctness)
