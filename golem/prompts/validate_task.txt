## Evaluation Criteria

1. **Task Completion**: Did the agent address all aspects of the original task?
2. **Quality**: Are the changes correct and reasonable for the task?
3. **No Regressions**: Did the agent avoid breaking existing functionality?
4. **Documentation**: Did the agent document its work appropriately?
5. **Verification Evidence**: Independent verification results are provided below.
   Do NOT check the event log for verification commands — the results below are
   from independent execution after the agent finished. Trust these results over
   any claims in the event log. If verification failed, the agent would have been
   retried — if you see this prompt, verification passed.
6. **Antipattern Detection**: Scan the git diff and event log for these
   antipatterns (ignore test code and debug-level logging):
   - *Traceback leaks*: Raw tracebacks or stack traces exposed in user-facing
     output, log messages, or API responses.
   - *Cross-module private access*: Code accessing ``_private`` attributes or
     methods of objects from other modules (violating encapsulation).
   - *String-matching control flow*: Using bare string comparisons
     (e.g., ``if status == "ready"``) for control flow instead of enums,
     constants, or dedicated types.
   - *Hardcoded UI states*: Inline styles or attributes that permanently hide
     or disable elements (e.g., ``display: none``, ``visibility: hidden``,
     ``hidden=True``, ``disabled=True``).
   - *Dead/unreachable code*: Statements placed after ``return``, ``raise``,
     or ``sys.exit()`` at the same indentation level (unreachable at runtime).
   - *Empty exception handlers*: ``except`` clauses whose body is only
     ``pass`` or ``...`` (silently swallowing errors).
   - *I/O calls without error handling*: File, network, or subprocess calls
     (``open``, ``requests.get``, ``subprocess.run``, etc.) not wrapped in a
     ``try/except`` block — verify that error handling exists nearby.
   - *Data contract mismatches*: camelCase dict key access in Python code
     (suggesting frontend/backend naming convention mismatch) or direct
     ``.json()["key"]`` / ``.json().get("key")`` access on API responses
     without type validation — verify keys match the backend response shape.
   If any antipattern is detected in a code_change task, flag each in concerns.
7. **Cross-Module Consistency**: When the diff adds or modifies code that
   reads from shared data structures (dicts, JSON, dataclass fields):
   - Open golem/types.py (included below) and verify the keys used match
     the TypedDict contract
   - If the code accesses dict keys not defined in types.py, flag as HIGH concern
   - If test fixtures construct dicts, verify their shape matches types.py
   - If new cross-module dict shapes should be in types.py but aren't, flag as concern
8. **Test Validity**: Tests must verify behavior, not just confirm assumptions:
   - Check that test fixtures use realistic data shapes (matching types.py contracts)
   - Flag tests where mocks/fixtures mirror the implementation rather than the
     contract (circular validation — the bug class this criterion prevents)
   - For cross-module features, verify at least one test exercises the real
     producer -> consumer path end-to-end
9. **Specification Fidelity** (when specs are present in the agent's report):
    If the agent's completion report includes `specs_satisfied`, verify each claim
    by checking the diff. Specs that are claimed as satisfied but not evidenced in
    the diff should be flagged in concerns.

9. **Reproduction Test** (bug-fix tasks only): If the task subject contains words
   like "fix", "bug", "error", "crash", "regression", or "broken", check the diff
   for at least one new test function. Bug fixes without reproduction tests have
   lower confidence that the fix addresses the actual issue. If no reproduction test
   is found, flag in concerns and lower confidence by 0.10.

Rules:
- **verdict**: One of "PASS", "PARTIAL", or "FAIL"
  - PASS: The task was fully completed with acceptable quality
  - PARTIAL: The task was partially completed; specific aspects need follow-up
  - FAIL: The task was not meaningfully completed or introduced problems
- **confidence**: REQUIRED, 0.0 to 1.0 — how confident you are in the verdict.
  Must reflect how much evidence you had to make the judgment:
  - 0.9-1.0: You can see all expected changes, tests, and they look correct
  - 0.7-0.89: Changes look correct but you cannot fully verify (e.g. diff was truncated)
  - 0.5-0.69: Partial evidence — some expected items present, others unclear
  - Below 0.5: Low evidence — verdict is largely a guess
- **concerns**: List specific, actionable items (empty list if none)
- **files_to_fix**: List specific file paths that need modification (empty list if none)
- **test_failures**: List specific failing test names or error messages from the event log (empty list if none)
- **task_type**: One of "code_change", "investigation", "documentation", "configuration", "other"

Evaluation notes:
- If there were no code changes and the task was investigative, evaluate based on documentation and tools used
- If there is a git diff, evaluate the quality and relevance of the changes
- If the agent encountered errors, weigh whether they were resolved or left unaddressed
- An empty git diff is fine for investigation/documentation tasks
- If the git diff is truncated, lower your confidence accordingly and note it in concerns
- If the independent verification results below show failures, flag them in concerns and lower confidence
- If antipatterns are detected in a code_change task, lower confidence by 0.05 per antipattern found, up to a maximum of 0.15, and flag each in concerns
