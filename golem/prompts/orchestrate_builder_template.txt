## Specification
[All SPEC statements from Phase 2]

## Context from exploration
[Phase 1 findings - file structures, patterns, key code snippets]

## Prior build output (if this is Builder #2+)
[Summary from previous Builder: files created, key decisions, test patterns]

## Your task
[Specific instructions: what to implement, which files to modify]

## Scope boundaries
Allowed actions:
- Implement exactly what the specification requires
- Add/modify tests for the changed code
- Add type annotations to new or changed code
- Fix imports needed for your changes

Prohibited actions:
- Refactor code unrelated to the task
- Change architecture or project structure
- Add features beyond the specification
- Modify CI/CD configuration or build scripts
- "Improve" surrounding code that works correctly

If you discover a pre-existing bug unrelated to your task, note it in your
report but do NOT fix it.

## Edge case checklist
Verify your implementation handles these categories where applicable:
- Null/None inputs and empty collections
- Boundary values (0, -1, max int, empty string)
- Special characters in string inputs
- Concurrent access / race conditions (if async or multi-threaded)
- Large inputs (performance-sensitive paths)

## Self-verification
Before reporting back, run ONLY these - nothing more:
1. pytest path/to/your/test_file.py -x  (targeted, NOT full suite)
2. black --check path/to/changed/files
3. pylint --errors-only path/to/changed/files
Do NOT run pytest --cov, pytest without a path, or the full test suite.
The Verifier handles that.
